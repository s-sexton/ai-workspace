# Jira Report

The first assistant report is generated by `assistant.src.jira_report`.

## Scope

The report generator:

-   Accepts normalized `JiraIssue` objects
-   Produces Markdown
-   Summarizes total issue count
-   Summarizes issue counts by status and priority
-   Includes a Markdown table of issue details
-   Escapes Markdown table values

It does not read configuration, call Jira, write files, use an LLM, or schedule
automation.

## Public API

``` python
generate_jira_report(issues, generated_at, timezone_name="America/Chicago")
```

`generated_at` is passed in by the caller so tests and reports remain
deterministic.

## Layer Boundary

The report generator depends on normalized issue objects from `common.jira`.

The intended flow is:

``` text
config -> Jira client -> normalized Jira issues -> Markdown report
```

## Local Runner

`assistant.src.run_jira_report` wires the current layers together with a fake
Jira transport:

``` text
load config -> static Jira response -> normalize issues -> generate Markdown -> write reports/jira-report.md
```

Run it from the workspace root:

``` powershell
python -m assistant.src.run_jira_report
```

This runner is for local infrastructure testing only. It does not call live
Jira, does not require real credentials, and does not write to Jira.
